# **Gitコミットメッセージ運用ガイド**

本ドキュメントは、開発現場での判断を迅速化するため、設計思想に基づき、従来のリストを7つに厳選して統合したものである。

## **1\. 設計思想（Why）**

接頭辞を使い分ける目的は、変更がシステムの外側（ユーザー）と、内側（コードや環境）のどちらを向いているかを明確にすることにある。これにより、履歴の検索性が向上し、修正による影響範囲を即座に判別することが可能となる。

## **2\. 接頭辞（プリフィックス）定義一覧**

実務においては、以下の7種を適切に選択することで、すべての作業を網羅する。

| 接頭辞 | 対象範囲 | 変更の性質 | 具体的な内容例 |
| :---- | :---- | :---- | :---- |
| **feat** | ユーザー | 機能追加・変更 | 新機能の実装、既存機能の仕様変更、UIの更新 |
| **fix** | ユーザー | 不具合修正 | バグの改修、タイポの修正（表示に影響するもの） |
| **refactor** | 内部構造 | 構造改善 | ロジックの整理、変数名の変更、冗長なコードの排除 |
| **docs** | 文書 | ドキュメント | READMEの更新、コード内コメントの加筆修正 |
| **style** | 内部品質 | 書式調整 | インデントや改行の調整、セミコロンの付与 |
| **test** | 内部品質 | テスト関連 | テストコードの新規作成、既存テストの修正 |
| **chore** | 環境/雑務 | その他 | ライブラリ更新、ビルド設定変更、不要ファイルの削除 |

## **3\. 分類に迷った際の判断マトリクス**

「なんとも言えない変更」に直面した際は、以下の優先順位に従って分類を決定する。

| 判断軸 | 判定基準 | 推奨される接頭辞 |
| :---- | :---- | :---- |
| **ユーザー認識** | 挙動の変化により、ユーザー体験がアップデートされるか | **feat** |
| **内部品質向上** | 挙動は変わらず、コードの可読性や保守性が向上したか | **refactor** |
| **環境維持管理** | 機能開発ではなく、ツールの設定やファイルの整理か | **chore** |
| **文書・書式** | 挙動に一切影響せず、テキストや見た目の形式のみか | **docs** / **style** |

## **4\. 記述のガイドライン（Format）**

コミットメッセージは、簡潔かつ正確な記述を旨とする。

| 項目 | 記述規則 | 留意点 |
| :---- | :---- | :---- |
| **1行目（要約）** | 20〜30文字程度 | 句読点を用いず、接頭辞の後に半角スペースを入れる |
| **2行目（空行）** | 必ず空行とする | ツール上の可読性を確保するために必須 |
| **3行目以降（詳細）** | 箇条書きを推奨 | 接続詞の重複を避け、論理関係を明確にする |
| **記号の使用** | 最小限に留める | 引用時のみ使用し、目立たせるための多用は避ける |

### **記述例**

feat: ユーザー検索機能にフィルタオプションを追加

\- 検索結果に日付範囲指定による絞り込み機能を追加  
\- UI上の日付選択カレンダーコンポーネントを実装  
\- 検索パラメータのバリデーションロジックを強化
